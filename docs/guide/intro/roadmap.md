# 开发地图

| 层级       | 含义                               |
| ---------- | ---------------------------------- |
| **主题**   | 长期方向（为什么做）               |
| **能力块** | 一组可以独立验收的能力（做什么）   |
| **交付项** | 一条能做完、能验证的事（怎么验收） |

**状态：** ✅ 已完成 · 🚧 进行中或暂缓 · 📝 尚未排期或待拆解

表中「交付项」尽量同一写法：**短名 + 括号内一句**，避免一行过长或过短。

## 当前执行顺序

1. **🚧 ledger 第一版**（1～2 天，做完即放下）— 文本识别 + 复制导出，一种目标格式即可  
2. **收费 + 推广底座** — 云会员、定价页、老用户通知  
3. **结构化指标 schema** → chart 改版、A 上下文出口  
4. **IndexedDB 迁移**（大项，与 3 设计并行，不挡 2）  
5. 其余按下方 2.0 愿景清单排期

### 主题：AI 与数据分析

下列按 **A → C → B** 顺序推进：Pomotention 不做聊天主场，而是给用户的**主力 AI**（Kimi / Claude / Cursor 等）喂上下文、提供工具、在 App 内展示写回结果。

| 链路 | 含义 | 是否需要 MCP |
| :--: | ---- | :------------: |
| **A** | 上下文出口：结构化摘要导出给外部 AI | 否 |
| **C** | 工具层：AI 读写 Pomotention（MCP Server 等） | 是 |
| **B** | 觉察卡片：App 内展示 AI 写回的内容（非聊天框） | 依赖 C |

#### 能力块：数据层（chart / A 的前置）

localStorage 已不适应当前数据量；chart 改版与上下文出口都依赖**稳定、可聚合**的本地读模型，故 storage 迁移单独立项、优先于 A 的完整形态。

| 状态 | 交付项 | 备注 |
| :--: | ------ | ---- |
|  📝  | 存储迁移（localStorage → IndexedDB） | 大改造；迁移策略、与 Supabase 同步的衔接需单独设计 |
|  📝  | 结构化指标 schema（chart 与 A 共用） | 统一定义周/Activity 级聚合字段，避免 chart 与导出各算各的 |
|  📝  | 记账/ledger 字段（可选扩展） | 与「记账系统兼容」共用：解析后的 posting 摘要可进 A 出口，不做账套管理 |

### 主题：记账系统兼容（收尾）

与 Task 追踪、时间记录并列：**在 Pomotention 里快记，在外部 plaintext 账本里入账**。不做余额、报表、多账套；只做「写 → 识别 → 导出」。

| 状态 | 交付项 | 备注 |
| :--: | ------ | ---- |
|  🚧  | 书写与快记基础设施 | `useTaskRecordShortcuts`、`StateLogModal`、`recordedAt` 等（见 1.0） |
|  📝  | ledger 语法识别器（初版） | **当前 sprint**；beancount 或 hledger 二选一 |
|  📝  | 一键复制 ledger 片段 | **当前 sprint**；与识别器同批 |
|  📝  | Task 内 code block / 专用模板 | 可第二版；第一版可只靠现有书写区 |
|  📝  | 与 IndexedDB 迁移的衔接 | 若存解析结果或历史导出记录，随 storage 迁移一并设计 |

**边界：** 不替代 Beancount/Fava、hledger-web 等；合规与税表不在范围内。

### 主题：一起做感（无实时沟通）

不做聊天、语音房间或实时 IM（合规与实现成本高）。仅在界面轻量提示「此刻/近日有人在专注」，让用户感到并非独自做事。
